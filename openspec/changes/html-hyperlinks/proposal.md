# Proposal: HTML-like hyperlinks for text with `link`

## Чому

- Користувачі вводять markdown-посилання `[label](url)` у текст; після сабміту зберігаються **видимий текст** `label` і властивість **`link`**.
- На canvas текст досі виглядає як звичайний, а клік здебільшого очікується біля **іконки**, а не на підписі — це слабко відповідає очікуванню «гіперпосилання як у HTML» (issue #11024 / `docs/specs/11024.md`).

## Що

1. **Візуально**: для текстових елементів з `element.link` — колір типовий для посилань (наприклад палітра `blue[4]` / `#1971c2`, узгоджений з іконками hyperlink) і **підкреслення** по рядках на canvas; у SVG експорті — відповідний `fill` + `text-decoration: underline` на `<text>` (зовнішній `<a>` вже може існувати).
2. **Взаємодія**: для **тексту** з `link`, коли елемент **не вибраний**, клік по **bounding box** тексту (на десктопі) вважається кліком по посиланні — відкриття URL у новій вкладці з тими ж правилами, що й для існуючого `handleElementLinkClick` (`_blank` для зовнішніх URL, `onLinkOpen` тощо).
3. **Без зміни**: повна markdown-розмітка; поведінка embeddable / інших типів елементів з `link`; UX **редагування** тексту (wysiwyg) — лише рендер і hit-test поза режимом редагування тексту.

## Вплив

| Область | Ризик / примітка |
|---------|------------------|
| `packages/element/src/renderElement.ts` | Зміна пікселів кешу тексту; потрібна узгодженість з темою (dark). |
| `packages/excalidraw/components/hyperlink/helpers.ts` | Розширення `isPointHittingLink`: текст з посиланням стає «клікабельним» по всьому bbox — вибір елемента одним кліком по тексту ускладнюється (альтернатива: marquee, клік поза текстом). |
| `staticSvgScene.ts` | Експортований SVG виглядає ближче до семантики посилання. |
| Тести | Можливі оновлення снапшотів / нові unit-тести на helpers або парсер (якщо додаються). |
